## 📋 第四章: 开发流程规范

### 4.1 Session驱动的原子开发流程

```
每次开发Session必须遵循:

Step 1: 启动时读取 SESSION_START.md
        ↓
Step 2: 执行 V-1~V-4 验证 (脚本/配置/文件/日志)
        ↓
Step 3: 从Task List选择下一个待办任务
        ↓
Step 4: 实现任务 (写代码/脚本/配置)
        ↓
Step 5: 运行质量门禁检查
        ├─ 通过? → 继续
        └─ 不通过? → 修复后再提交
        ↓
Step 6: 更新SESSION_START.md (记录本次变更)
        ↓
Step 7: 更新COMPLETENESS_MATRIX进度
        ↓
Step 8: (可选) 创建Git commit或备份快照
```

### 4.2 文件命名规范

```
# 配置文件
config.yaml                    ← 主配置 (YAML格式)
schedule-config.yaml           ← 调度配置
data-classification.yaml       ← 数据分类规则

# Skill脚本
business-email-writer/generate.ts    ← TypeScript (如有复杂逻辑)
email-scheduler/dispatcher.ps1      ← PowerShell (Windows系统集成)
inbox-monitor/checker.js            ← Node.js (跨平台能力)

# 数据文件
clients/by-email-account/account-001/client-list.json
products/categories/rc-toys/items/product-001.json
.memory/working/today.json
.memory/episodic/emails/2026-05/sent-batch-001.json

# 日志文件
.logs/daily/2026-05-09.log
.logs/errors/error-2026-05-09.log
.security/logs/security-audit.log
```

### 4.3 Commit/备份规范

```
格式: [type] scope: description

type: feat/fix/docs/config/security/refactor/test/chore
scope: skill名/模块名 (如: email-writer, scheduler, memory)
description: 简短描述 (中文, ≤50字)

示例:
feat(email-writer): 新增多语言邮件模板支持
fix(scheduler): 修复时区映射表缺少阿根廷问题
security(accounts): 升级密码加密算法至AES-256-GCM
docs(readme): 更新工作区目录结构说明
```

### 5.3 禁止事项 (Anti-Patterns)

```
❌ 禁止: 将敏感信息(密码/API密钥)硬编码在脚本中 (必须用.env或Vault)
❌ 禁止: 在脚本中直接操作生产数据 (必须先备份)
❌ 禁止: 跳过错误处理直接try-catch空catch (必须有日志记录)
❌ 禁止: 使用console.log输出敏感信息 (用logger替代并脱敏)
❌ 禁止: 同一账号并发发送超过限制 (必须遵守速率限制)
❌ 禁止: 未加密存储客户联系方式 (必须字段级加密)
❌ 禁止: 忽略时区差异统一发送 (必须按客户当地时区)
❌ 禁止: 跳过用户直接发送重要客户邮件 (A级客户必须人工确认)
```

## 📋 第七章: 安全与合规

### 7.1 数据保护原则

```
原则1: 最小权限原则 (Principle of Least Privilege)
├─ 仅授权必要的文件访问权限
├─ 脚本以最低必要权限运行
└─ 定期审查权限列表

原则2: 数据最小化 (Data Minimization)
├─ 仅收集业务必需的数据
├─ 及时删除不再需要的信息
└─ 避免过度采集客户隐私

原则3: 默认加密 (Encryption by Default)
├─ 所有敏感数据默认AES-256加密
├─ 传输强制TLS/SSL
├─ 备份文件同样加密
└─ 密钥与数据分离存储

原则4: 审计可追溯 (Audit Trail)
├─ 所有操作记录不可篡改的日志
├─ 关键操作需留痕 (谁/何时/做了什么)
├─ 日志保留至少180天
└─ 异常行为自动告警
```

### 7.2 合规要求清单

```
□ GDPR合规 (如涉及欧盟客户):
  ├── 数据处理目的明确告知
  ├── 用户有权要求删除数据
  ├── 数据跨境传输有合法依据
  └── 发生 breach 72小时内通知

□ 中国网络安全法:
  ├── 重要数据本地化存储 ✓ (已满足)
  ├── 日志留存≥6个月
  ├── 网络安全等级保护备案
  └── 个人信息去标识化处理

□ 邮件营销法规:
  ├── CAN-SPAM Act (美国)
  ├── GDPR电子营销条款 (欧盟)
  ├── CASL (加拿大)
  └── 提供退订选项 (即使B2B也建议)
```

```

├─ 触发条件: SMTP返回550/552/421错误
├─ 自动响应: 暂停该账号→标记异常→通知用户
├─ 人工介入: 登录网页版解封或联系客服

├─ 触发条件: 用户连续3次大幅修改同类型草稿(>70%修改量)

├─ 人工介入: 查看蒸馏日志+手动微调规则
└─ 恢复流程: 下轮蒸馏验证改进效果

场景3: 数据文件损坏
├─ 触发条件: JSON解析失败/校验和不匹配
├─ 自动响应: 从 .security/backups/ 恢复最近版本
├─ 人工介入: 检查磁盘健康+文件系统错误
└─ 恢复流程: 验证数据完整性后继续运行

场景4: 系统资源耗尽
├─ 触发条件: 内存>90% / CPU>95% / 磁盘<5%
├─ 自动响应: 暂停非关键Task→清理临时文件→释放缓存
├─ 人工介入: 排查泄漏进程/扩容硬件
└─ 恢复流程: 资源恢复正常后自动恢复Task
```
