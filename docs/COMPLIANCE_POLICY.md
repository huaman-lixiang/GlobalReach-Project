## 2. 数据分类与保护

### 2.1 数据分类矩阵

| 类别 | 示例字段 | 保存期限 | 加密要求 | 访问级别 |
|------|---------|---------|---------|---------|
| **个人身份信息(PII)** | 姓名、邮箱、电话 | 账户期内+90天 | AES-256 (静态+传输) | 用户本人+管理员 |
| **认证凭据** | 密码哈希、Token | Token有效期/密码变更 | SHA-256 (单向哈希) | 仅系统内部 |
| **业务数据** | 公司名、行业、客户列表 | 账户期内 | AES-256 (静态) | 用户本人+管理员 |
| **营销内容** | 邮件模板、活动配置 | 180天 | AES-256 (静态) | 用户本人 |
| **系统日志** | 审计日志、操作日志 | 365天 | SHA-256 (完整性校验) | 仅管理员 |
| **安全日志** | 登录失败、异常检测 | 730天 (2年) | SHA-256 (完整性校验) | 仅管理员+安全团队 |
| **错误日志** | 异常堆栈、错误详情 | 90天 | 无特殊要求 | 开发运维人员 |

### 2.2 数据敏感度标签

```
🔴 RESTRICTED (受限)   — 密码、Token、密钥 → 仅特定系统组件访问
🟠 CONFIDENTIAL (机密)  — PII、财务信息 → 需要明确授权
🟡 INTERNAL (内部)      — 业务数据、配置 → 组织内部可访问
🟢 PUBLIC (公开)        — 产品信息、帮助文档 → 所有人可访问
```

### 2.3 数据处理活动登记册

| 处理活动 | 数据类型 | 法律依据 | 保留期限 | 第三方 |
|---------|---------|---------|---------|--------|
| 用户注册/认证 | PII | 合同履行 | 账户期+90天 | 无 |
| 客户数据管理 | PII+业务数据 | 合同履行+合法利益 | 账户期内 | 无 |
| 邮件发送服务 | 邮件内容+元数据 | 用户同意/合同 | 180天 | 邮件服务商(Google/Microsoft等) |
| 营销活动分析 | 聚合统计数据 | 合法利益 | 365天 | 无 |
| 审计日志记录 | 操作元数据 | 法律义务 | 365天 | 无 |
| 安全监控 | IP地址、行为模式 | 合法利益 | 730天 | 安全服务(SIEM) |

## 3. 审计事件分类

### 3.1 必须审计的事件 (MUST AUDIT)

✅ **用户认证事件**
- 登录成功/失败 (`LOGIN` / `AUTH_FAILURE`)
- 登出 (`LOGOUT`)
- Token刷新 (`TOKEN_REFRESH`)
- 密码修改/重置 (`PASSWORD_CHANGE` / `PASSWORD_RESET`)
- 多因素认证启用/禁用

✅ **数据增删改操作 (CRUD)**
- 全部业务表的写操作:
  - Client: `CREATE/UPDATE/DELETE`
  - Campaign: `CREATE/UPDATE/DELETE`
  - Email: `CREATE/UPDATE/DELETE`
  - Account: `CREATE/UPDATE/DELETE`
  - User: `USER_CREATE/USER_UPDATE/USER_DELETE`

✅ **权限变更**
- 角色分配/撤销 (`ROLE_CHANGE`)
- 权限修改 (`PERMISSION_CHANGE`)
- 用户状态变更 (激活/停用/锁定)

✅ **系统配置变更**
- 系统参数修改 (`CONFIG_CHANGE`)
- 邮件模板更新
- 集成配置更改 (API密钥、Webhook)

✅ **批量操作**
- 批量导入 (`IMPORT_DATA`)
- 批量导出 (`EXPORT_DATA`)
- 批量删除 (`BULK_DELETE`)

✅ **异常/安全事件**
- 连续认证失败 (≥3次=锁定告警)
- 权限拒绝 (`ACCESS_DENIED`)
- 限流触发 (`RATE_LIMITED`)
- 可疑行为检测 (`SUSPICIOUS_ACTIVITY`)

### 3.2 建议审计的事件 (SHOULD AUDIT)

⚠️ **数据查询操作**
- 批量查询 (>100条记录)
- 全量导出请求
- 敏感字段访问 (密码、Token)

⚠️ **文件操作**
- 文件上传/下载
- 附件处理

⚠️ **其他**
- API密钥轮换
- 设置页面访问
- 隐私政策查看

### 3.3 审计日志字段定义

```javascript
{
  id: "BIGINT自增主键",
  userId: "UUID | 操作用户ID (可为null表示系统操作)",
  action: "STRING(100) | 操作类型枚举值",
  resourceType: "STRING(50) | 资源类型 (user/client/campaign/email/account/system)",
  resourceId: "STRING(100) | 资源ID (可为null)",
  ipAddress: "STRING(50) | 客户端IP地址",
  userAgent: "TEXT | 浏览器/客户端User-Agent",
  details: "TEXT(JSON) | 操作详情 (敏感字段自动脱敏)",
  severity: "ENUM | INFO/WARN/ERROR/CRITICAL",
  status: "ENUM | SUCCESS/FAILURE",
  sessionId: "STRING(255) | 会话ID (关联同一会话的操作链)",
  createdAt: "TIMESTAMP | ISO时间戳 (自动生成)"
}
```

### 3.4 严重级别定义

| 级别 | 触发条件 | 响应要求 | 示例 |
|-----|---------|---------|------|
| 🔴 CRITICAL | 安全漏洞利用、数据泄露 | ⚠️ 立即告警+人工介入 | 批量数据导出、权限提升攻击 |
| 🟠 ERROR | 操作失败、系统异常 | 🔔 自动告警+记录 | 数据库写入失败、认证系统故障 |
| 🟡 WARN | 可疑行为、策略违反 | 📝 记录+监控 | 连续登录失败、越权尝试 |
| 🟢 INFO | 正常业务操作 | 📊 统计分析 | 用户登录、数据创建/更新 |

## 4. 数据主体权利 (GDPR Art.15-22)

### 4.1 权利清单与实现

| 权利 | GDPR条款 | PIPL对应 | API端点 | 实现状态 |
|-----|---------|---------|---------|---------|
| **访问权** | Art.15 | 第44条 | `GET /api/v1/compliance/data-export` | ✅ 已实现 |
| **更正权** | Art.16 | 第46条 | `PUT /api/v1/users/profile` | ✅ 已实现 |
| **删除权 ("被遗忘权")** | Art.17 | 第47条 | `POST /api/v1/compliance/data-delete-request` | ✅ 已实现 (需二次确认) |
| **限制处理权** | Art.18 | 第48条 | `POST /api/v1/users/privacy-settings` | ⚠️ 部分实现 |
| **数据可携带权** | Art.20 | 第45条 | `GET /api/v1/compliance/data-export` (JSON格式) | ✅ 已实现 |
| **反对权** | Art.21 | 第49条 | `POST /api/v1/compliance/object` | ⚠️ 待完善 |
| **自动化决策相关权利** | Art.22 | 第24条 | N/A (当前不涉及) | ➖ 不适用 |

### 4.2 行使权利的流程

#### 数据访问权 (DSAR请求)

```
用户请求 → 身份验证 → 收集数据(最多30天) → 格式化输出(JSON) → 安全交付
```

**SLA**: 
- 简单请求: ≤ 7个工作日
- 复杂请求: ≤ 30天 (需提前通知延期)

**交付格式**:
- JSON (机器可读)
- 包含所有关联数据 (账户、客户、活动、邮件、审计轨迹)
- 不包含密码哈希、Token等敏感凭据

#### 数据删除权 (账户注销)

```
提交请求 → 二次确认(邮件验证) → 标记账户 → 
停止数据处理 → 匿名化/删除数据(90天内) → 发送完成确认
```

**例外情况** (法律允许保留):
- 法律诉讼相关证据
- 财务会计记录 (法定保存期)
- 已签署合同的必要数据

## 5. 保留与删除策略

### 5.1 自动删除策略

| 数据类型 | 触发条件 | 保留期 | 清理方式 | 备注 |
|---------|---------|-------|---------|------|
| 已注销用户数据 | 账户注销后 | 90天 | 自动匿名化 | PII字段替换为假名 |
| 未激活账户 | 注册后从未登录 | 30天 | 自动删除 | 发送提醒邮件后 |
| 过期Token | Token过期后 | 7天 | 自动清理 | RefreshToken表 |
| 临时上传文件 | 上传后 | 24小时 | 自动删除 | 导入/导出临时文件 |
| 缓存数据 | TTL到期 | 配置的时间 | 自动失效 | Redis缓存 |

### 5.2 手动审核保留

| 场景 | 条件 | 保留至 | 责任人 |
|-----|------|-------|--------|
| 法律诉讼 | 法院命令/律师函 | 案件结束+法定保存期 | 法务部 |
| 监管调查 | 监管机构要求 | 调查完成 | 合规官 |
| 内部审计 | 年度审计计划 | 审计报告发布 | 内审部 |
| 数据主体投诉 | 正式投诉流程 | 投诉解决 | DPO |

### 5.3 匿名化处理

**适用场景**: 分析用途、长期统计、研究目的

**处理方法**:
```javascript
// 匿名化示例 (PII字段)
{
  email: "user***@example.com",  // 部分遮蔽
  phone: "138****1234",           // 中间位遮蔽
  name: "用户A12345",             // 替换为伪名
  ipAddress: "192.168.*.*",       // 后两段遮蔽
  companyId: hash(companyId),     // 哈希替换
}
```

**不可逆性保证**:
- 使用单向哈希函数 (SHA-256)
- 添加随机盐值 (salt)
- 保留原始哈希映射表 (单独加密存储，定期销毁)

### 5.4 数据生命周期图

```
创建 → 活跃使用 → 非活跃(90天) → 归档(可选) → 匿名化/删除
  │        │           │              │            │
  ▼        ▼           ▼              ▼            ▼
收集    处理用途     最后访问      法律/业务需要   最终处置
同意    最小化原则   自动标记      手动审批       安全擦除
```

## 6. 技术控制措施

### 6.1 访问控制

**认证机制**:
- JWT (JSON Web Token) 短期访问令牌 (15分钟)
- Refresh Token 长期令牌 (7天，支持轮换)
- 支持MFA (多因素认证) - 待实现

**角色权限矩阵 (RBAC)**:

| 角色 | 用户管理 | 客户管理 | 活动管理 | 审计日志 | 系统配置 |
|-----|---------|---------|---------|---------|---------|
| ADMIN | ✅ 完全 | ✅ 完全 | ✅ 完全 | ✅ 查看+导出 | ✅ 完全 |
| USER | ❌ 仅自己 | ✅ 自己的 | ✅ 自己的 | ❌ 仅自己时间线 | ❌ |
| VIEWER | ❌ | 👁 只读 | 👁 只读 | ❌ | ❌ |

**资源所有权检查**:
- 非ADMIN用户只能访问自己创建的资源
- 强制 `userId` 字段匹配检查
- API层统一拦截 (RBAC中间件)

### 6.2 加密标准

| 数据状态 | 加密算法 | 密钥长度 | 实现位置 |
|---------|---------|---------|---------|
| 传输中 (TLS) | TLS 1.3 | ECDHE (P-256) | Nginx反向代理 |
| 静态存储 (PII) | AES-256-GCM | 256位 | PostgreSQL (pgcrypto) |
| 密码哈希 | bcrypt | cost=12 | 应用层 |
| Token签名 | HMAC-SHA256 | 256位 | JWT库 |
| 完整性校验 | SHA-256 | 256位 | 审计日志字段 |

### 6.3 日志与监控

**审计日志系统特性**:
- ✅ 写入后不可修改/删除 (append-only)
- ✅ 完整性保护 (哈希校验链)
- ✅ 时间戳同步 (NTP)
- ✅ 结构化格式 (JSON)
- ✅ 敏感数据自动脱敏

**集中式日志架构**:
```
应用服务器 → Loki (日志聚合) → Prometheus (指标) → Grafana (可视化)
                    ↓
              AlertManager (告警) → PagerDuty/邮件/Slack
```

**关键监控指标**:
- 每5分钟审计日志写入量
- 认证失败率 (>10%触发告警)
- 异常事件数量 (WARN/ERROR/CRITICAL)
- 数据导出请求频率
- 删除请求处理延迟

### 6.4 数据备份与恢复

**备份策略**:
- **实时复制**: PostgreSQL流复制 (主→从)
- **每日增量**: 00:00 UTC自动快照
- **每周全量**: 周日02:00 UTC完整备份
- **异地容灾**: 跨区域备份 (S3/Glacier)

**恢复目标**:
- RPO (恢复点目标): ≤ 1小时
- RTO (恢复时间目标): ≤ 4小时

**备份数据加密**:
- 传输加密: TLS 1.3
- 存储加密: AES-256
- 密钥管理: AWS KMS / HashiCorp Vault

## 7. 应急响应流程

### 7.1 数据泄露分级响应

| 等级 | 定义 | 响应时间 | 通报对象 |
|-----|------|---------|---------|
| 🔴 **严重 (Critical)** | >1000条PII泄露 或 包含密码/财务数据 | ≤ 1小时 | 监管机构+用户+管理层 |
| 🟠 **高 (High)** | 100-1000条PII泄露 | ≤ 4小时 | 用户+管理层 |
| 🟡 **中 (Medium)** | <100条PII泄露 或 系统日志泄露 | ≤ 24小时 | 受影响用户 |
| 🟢 **低 (Low)** | 非PII数据泄露 或 内部测试数据 | ≤ 72小时 | 内部记录 |

### 7.2 应急响应步骤

```
1. 发现与报告 (0-1h)
   ├── 识别泄露范围和类型
   ├── 初步遏制 (隔离受影响系统)
   └── 召集IRT ( Incident Response Team )

2. 调查与分析 (1-24h)
   ├── 取证 (保留日志、内存转储)
   ├── 根因分析 (RCA)
   └── 影响评估 (受影响用户名单)

3. 制止与修复 (24-72h)
   ├── 修补漏洞
   ├── 强化控制措施
   └── 监控防止复发

4. 通知与沟通 (按等级要求)
   ├── 监管通报 (GDPR: 72小时内)
   ├── 用户通知 (清晰、及时)
   └── 媒体应对 (必要时)

5. 恢复与复盘 (72h+)
   ├── 服务恢复验证
   ├── 事后复盘 (Post-mortem)
   └── 流程改进
```
