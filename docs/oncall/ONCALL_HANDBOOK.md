## 第一章：On-call 制度概述

### 1.1 定义与目的

#### 1.1.1 什么是 On-call

**On-call（值班）** 是一种 7×24 小时系统保障制度，确保 GlobalReach V2.0 平台在任何时间点都有具备技术能力的工程师能够：

- **接收并响应** 系统产生的告警通知
- **快速诊断** 故障根因，缩小影响范围
- **执行修复** 操作，恢复服务可用性
- **协调资源** 在需要时升级给更高级别的支持人员
- **记录事件** 为后续复盘和知识积累提供数据支撑

#### 1.1.2 制度目的

| 目标维度 | 具体指标 | 衡量方式 |
|---------|---------|---------|
| **可用性保障** | 平台可用性 ≥ 99.9% | Uptime 监控 / Status Page |
| **响应速度** | P0 告警 < 5 分钟确认 | AlertManager 确认时间戳 |
| **修复效率** | 平均 MTTR < 30 分钟 | 事件关闭时间统计 |
| **覆盖完整性** | 7×24 无盲区 | 排班表连续性检查 |
| **知识传承** | 每次事件均有记录 | Post-Mortem 完成率 |

### 1.2 角色定义

#### 1.2.1 Primary（主值班）

**职责范围**：
- 所有告警的第一接收人和主要响应者
- 负责初步诊断、判定严重级别、执行标准修复动作
- 在能力范围内独立处理事件
- 维护当前班次的值班日志
- 在需要时发起升级

**能力要求**：
- 熟悉 GlobalReach 全部核心组件（API/PostgreSQL/Redis/Nginx）
- 能够独立使用 Runbook（RB-001~007）执行标准操作
- 熟练使用决策树（TT-001~006）进行故障定位
- 了解故障模式库（FMB）中的 22 个常见故障模式
- 具备基本的 Linux/Docker 操作能力

**值班期间行为规范**：
- 保持通讯工具在线且可被即时触达
- 不在无备份的情况下离开工位超过 15 分钟
- 每小时至少查看一次监控 Dashboard
- 任何不确定的操作先查阅文档再执行
- 遇到无法解决的问题在 SLA 时限内及时升级

#### 1.2.2 Secondary（备值班）

**职责范围**：
- 作为 Primary 的备份，在 Primary 无法响应时接管
- 协助 Primary 处理复杂或大规模事件
- 参与重大事件的联合诊断
- 在 Primary 升级时作为第一级升级目标

**能力要求**：
- 具备与 Primary 相当的技术能力
- 对系统有全局视角的理解
- 有处理过至少 5 次 P0/P1 事件的经验

**待命期间行为规范**：
- 保持通讯工具在线（可设为免打扰但需能被紧急呼叫唤醒）
- 了解当前 Primary 正在处理的活跃事件
- 准备好随时接手值班职责
- 每日晨会了解前一班的交接情况

#### 1.2.3 Escalation Contact（升级联系人）

**分级定义**：

| 级别 | 角色 | 触发条件 | 响应期望 |
|------|------|---------|---------|
| **L1** | Secondary | Primary 超时未响应 / Primary 请求协助 | < 10 分钟 |
| **L2** | Tech Lead / Senior SRE | P0 事件 30 分钟未解决 / 需要架构决策 | < 15 分钟 |
| **L3** | Engineering Manager | 影响范围扩大 / 需要外部协调 / SLA 即将违约 | < 30 分钟 |
| **L4** | VP Engineering / CTO | 全平台宕机 / 数据安全事件 / 法律合规风险 | 立即 |
| **L5** | CEO + 法务/PR | 重大公关危机 / 用户数据泄露 / 监管机构介入 | 立即 |

### 1.3 轮值规则

#### 1.3.1 基本轮值模式：周轮换

```
时间线示例（4 人团队）:

第 1 周:
  Mon 00:00 ~ Sun 23:59  ┃─ Primary: Alice  ┃─ Secondary: Bob
第 2 周:
  Mon 00:00 ~ Sun 23:59  ┃─ Primary: Bob    ┃─ Secondary: Carol
第 3 周:
  Mon 00:00 ~ Sun 23:59  ┃─ Primary: Carol  ┃─ Secondary: Dave
第 4 周:
  Mon 00:00 ~ Sun 23:59  ┃─ Primary: Dave   ┃─ Secondary: Alice
... 循环
```

**规则细节**：
- 每周周一 00:00 自动切换值班人
- Primary 和 Secondary 不能是同一人
- 切换时刻通过自动化脚本 (`scripts/oncall-manager.sh --handover`) 执行
- 切换后自动发送通知到团队频道

#### 1.3.2 技能分配原则

不同技能背景的人员应合理分配到不同的时间段：

| 时间段 | 特征 | 推荐 Primary 类型 | 理由 |
|--------|------|------------------|------|
| 工作日白天 (09:00-18:00) | 高业务量、可获取充分支持 | 全栈工程师 | 业务高峰期需要全面能力 |
| 工作日晚间 (18:00-24:00) | 中等业务量、部分支持离线 | 后端/运维工程师 | API 和基础设施问题居多 |
| 夜间/凌晨 (00:00-09:00) | 低业务量、需独立决策 | 资深 SRE | 独立处理能力最重要 |
| 周末全天 | 低业务量但支持最少 | 轮流分配 | 公平性优先 |

#### 1.3.3 避免连续规则

为防止值班疲劳，以下约束必须满足：

```yaml
# 轮值约束配置
constraints:
  # 同一人不能连续担任 Primary
  no_consecutive_primary: true

  # 同一人担任 Primary 后，下一周至少间隔一周才能再次担任 Primary
  primary_cooldown_weeks: 1

  # 同一人不能同时担任 Primary 和 Secondary
  no_simultaneous_roles: true

  # 连续两周值班（Primary 或 Secondary）后必须休息一周
  max_consecutive_oncall_weeks: 2

  # 节假日期间优先安排有经验的成员
  holiday_prefer_seniority: true

  # 新成员（入职 < 3个月）必须在有经验成员陪同下值班
  junior_requires_shadowing: true
```

#### 1.3.4 特殊情况处理

| 场景 | 处理方案 |
|------|---------|
| 值班人请假/病假 | 提前 48 小时通知，由排班协调员调整 |
| 值班人临时不可用 | Secondary 自动升任 Primary，从 Escalation L2 中指定新 Secondary |
| 团队扩容/缩容 | 更新排班池，重新计算轮值顺序 |
| 重大发布窗口 | 可安排双 Primary（发布负责人 + 当周值班人） |
| 节假日 | 提前 2 周公布节假日排班表 |

### 1.4 SLA 响应时间要求

#### 1.4.1 分级 SLA 定义

| 严重级别 | 名称 | 响应时限 | 修复时限 | 影响范围 | 示例 |
|---------|------|---------|---------|---------|------|
| **P0** | Critical | **< 5 分钟** | **< 30 分钟** | 全平台不可用 / 数据风险 | API 宕机、DB 主库崩溃、安全入侵 |
| **P1** | High | **< 15 分钟** | **< 2 小时** | 核心功能受损 / 大量用户受影响 | Redis 故障、Nginx 502、队列积压 |
| **P2** | Medium | **< 1 小时** | **< 8 小时** | 非核心功能异常 / 少数用户受影响 | 单个 Campaign 发送慢、报表延迟 |
| **P3** | Low | **< 4 小时** | **< 48 小时** | 轻微问题 / 仅内部可见 | 日志警告、非关键 metrics 异常 |

#### 1.4.2 SLA 计算规则

```
SLA 时间计算起点 = 告警首次触发时间（AlertManager timestamp）
SLA 时间计算终点 = 人工确认时间（ACK timestamp）/ 事件解决时间（Resolve timestamp）

注意:
- 不包含告警降噪/聚合的处理时间
- "响应" = 人在系统中 ACK 了该告警
- "修复" = 服务恢复正常且验证通过
- 升级等待时间计入总 MTTR
```

#### 1.4.3 SLA 违约后果

| 违约类型 | 后果 | 改进措施 |
|---------|------|---------|
| P0 响应超时 (>5min) | 自动升级至 L2 + 管理层通知 | 分析超时原因，优化通知链路 |
| P0 修复超时 (>30min) | 启动 Major Incident 流程 | 事后必须做 Full Post-Mortem |
| P1 响应超时 (>15min) | 自动升级至 L1 (Secondary) | 检查值班人可达性 |
| 累计月违约率 > 5% | 团队级复盘 + 流程审查 | 可能需要增加值班人力 |
| 个人累计违约 > 3次/月 | 一对一沟通 + 培训需求评估 | 安排 shadowing 学习 |

#### 1.4.4 SLA 豁免条件

以下情况不计入 SLA 违约：
- 计划内维护窗口期间的预期性告警（已在维护日历中登记）
- 第三方依赖故障（如 SMTP 提供商宕机）— 但仍需在豁免时间内响应
- 不可抗力（自然灾害、数据中心断电）
- 已知 Bug 的重复告警（已在 Known Issues 中标记）

## 第二章：告警分级与处理 SOP

### 2.1 基于 AIOps 的告警分类体系

本系统的告警分级基于 O01（AIOps 智能告警降噪）模块输出的标准化告警。AIOps 引擎会对原始告警进行关联分析、降噪聚合后，输出结构化的告警事件给 On-call 系统。

#### 2.1.1 告警来源映射

| AIOps 输出字段 | On-call 使用方式 | 说明 |
|---------------|-----------------|------|
| `severity` | 直接映射到 P0-P3 | critical→P0, warning→P1/P2, info→P3 |
| `clusterId` | 关联同一根因的所有告警 | 用于批量处理 |
| `rootCause` | 优先处理根因告警 | 避免在症状上浪费时间 |
| `suggestedAction` | 推荐的自愈动作级别 | L1-L5 对应不同审批流程 |
| `suppressionDecision` | notify/suppress/aggregate | suppressed 的不需要人工处理 |

#### 2.1.2 告警分级决策矩阵

```
收到 AIOps 输出的告警事件
        │
        ▼
  ┌─────────────────┐
  │ severity = ?    │
  └────┬────┬───────┘
       │    │
  critical  warning/info
       │    │
       ▼    ▼
  ┌─────────┐  ┌─────────────────────┐
  │ 影响    │  │ 持续时间?            │
  │ 范围?   │  │                     │
  └──┬──┬──┘  └──┬──────────┬───────┘
     │  │        │          │
  全部 核心   >30min    <30min
  服务 功能     │          │
     │  │        ▼          ▼
     ▼  ▼     P1-High    P2-Medium
   P0   P0
     │
     ▼
  P0-Critical
  (全平台紧急)
```

### 2.2 P0-Critical 级别 SOP

#### 2.2.1 标准 T+n 流程

```
T+0min   ┃─ 🚨 告警触发 → 电话/SMS/IM 三通道同时推送
T+2min   ┃─ 📞 Primary 接收 → 立即 ACK（在 AlertManager/Grafana 中确认）
T+5min   ┃─ 🔍 初步诊断 → 使用 TT 决策树确定方向
             ├─ 检查 docker ps（所有容器状态）
             ├─ 检查 Prometheus targets（采集状态）
             └─ 查看 Loki 最近错误日志
T+10min  ┃─ 📋 判定根因 → 对照 FMB 匹配已知模式
             ├─ 匹配成功 → 按 RB 执行标准修复步骤
             └─ 未匹配 → 进入探索模式（收集更多信息）
T+15min  ┃─ ⚡ 实施修复 → 执行最高效的恢复动作
             ├─ 优先恢复服务可用性（先止血）
             └─ 根因修复可以稍后进行
T+25min  ┃─ ✅ 效果验证 → O03 巡检引擎确认恢复
             ├─ health check 通过
             ├─ 关键指标回到正常基线
             └─ 用户侧抽样验证
T+30min  ┃─ 📝 事件关闭/升级 → 如果未恢复则立即升级
```

#### 2.2.2 每步详细动作

**Step 1: 接收与确认（T+0 ~ T+2）**

| 动作项 | 具体操作 | 工具/命令 | 预期结果 | 超时处理 |
|-------|---------|----------|---------|---------|
| 接收通知 | 接听电话/查看 IM | 手机/钉钉/Slack | 知晓告警内容 | >2min 自动升级到 Secondary |
| 安全确认 | 确认本人是当前 Primary | 查看 oncall schedule | 确认身份 | 如非本人，转发给正确的人 |
| ACK 告警 | 在 AlertManager 中确认 | Grafana AlertManager UI / API | 告警状态变为 pending | >5min 触发自动升级 |
| 初步通报 | 发送"已接手"消息到频道 | IM group message | 团队知晓有人处理 | - |

**Step 2: 诊断定位（T+2 ~ T+10）**

| 动作项 | 具体操作 | 工具/命令 | 预期结果 | 超时处理 |
|-------|---------|----------|---------|---------|
| 容器状态检查 | 查看所有服务容器 | `docker ps -a` | 识别异常容器 | - |
| 资源使用检查 | CPU/内存/磁盘 | `docker stats --no-stream` | 发现瓶颈 | - |
| 日志快速扫描 | 最近错误日志 | Loki LogQL / `docker logs --tail 100` | 错误模式识别 | - |
| 依赖链检查 | DB/Redis/Nginx 连通性 | `docker exec ... pg_isready` / `redis-cli ping` | 定位故障层级 | - |
| 决策树导航 | 选择对应 TT | TT-001 ~ TT-006 | 缩小排查范围 | >8min 请求 Secondary 协助 |

**Step 3: 处理与修复（T+10 ~ T+25）**

| 动作项 | 具体操作 | 工具/命令 | 预期结果 | 超时处理 |
|-------|---------|----------|---------|---------|
| 选择修复策略 | 根据诊断选择 RB 场景 | Runbook RB-001~007 | 明确操作步骤 | - |
| 执行修复操作 | 按步骤执行 | Docker CLI / API / Script | 服务开始恢复 | - |
| 风险评估 | 评估操作的副作用 | 经验判断 + 文档参考 | 确认安全性 | 有疑问则升级 |
| 变更记录 | 记录所有操作 | 事件 Timeline | 审计追踪 | - |

**Step 4: 验证与关闭（T+25 ~ T+30）**

| 动作项 | 具体操作 | 工具/命令 | 预期结果 | 超时处理 |
|-------|---------|----------|---------|---------|
| 健康检查 | 端到端健康验证 | `curl /health` / O03 巡检 | 所有 check 通过 | - |
| 指标回归 | 确认关键指标正常 | Grafana Dashboard | 回到基线范围 | - |
| 用户验证 | 抽样用户功能测试 | 手动测试核心流程 | 用户可正常使用 | - |
| 事件关闭 | 更新事件状态为 resolved | Incident API / UI | 事件归档 | - |

```
║    API 挂了?     → RB-001 场景 1/2/3                      ║
║    DB 挂了?      → RB-002 场景 1/2                         ║
║    Redis 挂了?   → RB-003 场景 1                           ║
║    Nginx 挂了?   → RB-004 场景 1                           ║
║    磁盘满了?     → RB-006 场景 4                           ║
║  第四步: 先恢复服务，再查根因                               ║
║  第五步: O03 巡检验证 → 关闭事件                            ║
╚══════════════════════════════════════════════════════════╝
```

### 2.3 P1-High 级别 SOP

#### 2.3.1 标准 T+n 流程

```
T+0min    ┃─ 🔔 告警触发 → IM + Email 推送（不打电话）
T+10min   ┃─ 👀 Primary 确认 → ACK 告警
T+15min   ┃─ 🔍 诊断定位 → 使用 TT 缩小范围
T+30min   ┃─ 📋 制定修复计划 → 评估影响和风险
T+60min   ┃─ ⚡ 实施修复 → 执行 Runbook 操作
T+90min   ┃─ ✅ 验证效果 → 确认功能恢复
T+120min  ┃─ 📝 记录事件 → 更新 timeline
```

#### 2.3.2 P1 与 P0 的关键差异

| 维度 | P0-Critical | P1-High |
|------|------------|---------|
| 通知方式 | 电话+SMS+IM（三通道） | IM + Email（双通道） |
| 响应时限 | 5 分钟 | 15 分钟 |
| 修复时限 | 30 分钟 | 2 小时 |
| 是否允许分步修复 | 先止血再根治 | 可以计划性修复 |
| 是否需要实时通报 | 每 10 分钟更新一次 | 每小时更新一次 |
| 是否必须 Post-Mortem | 必须 | 事件影响大则建议 |

### 2.4 P2-Medium 级别 SOP

#### 2.4.1 标准 T+n 流程

```
T+0min    ┃─ 📧 告警触发 → Email 推送 + IM 频道消息
T+30min   ┃─ 👀 Primary 确认 → ACK（可在方便时处理）
T+60min   ┃─ 🔍 初步评估 → 判断是否需要立即处理
T+4h      ┃─ 📋 如果需要修复 → 排入当前任务队列
T+8h      ┃─ ⚡ 完成修复 → 或给出明确的排期
T+8h+     ┃─ 📝 关闭事件 → 记录处理结果
```

#### 2.4.2 P2 处理策略

P2 级别事件的特点是"不影响核心业务但需要关注"，因此处理策略更加灵活：

- **可以在当前值班期间择机处理**，不必中断其他工作
- **如果正在处理 P0/P1 事件**，P2 可以暂时搁置（但需要在 1 小时内 ACK）
- **如果多个 P2 同时出现**，按影响用户数量排序处理
- **P2 事件通常不需要紧急升级**，但如果 8 小时内未处理应升级到 L1

### 2.5 P3-Low 级别 SOP

#### 2.5.1 标准 T+n 流程

```
T+0min    ┃─ 📋 告警触发 → Email 每日汇总
T+4h      ┃─ 👀 浏览确认 → ACK（可在日常工作中处理）
T+24h     ┃─ 📋 评估是否需要修复 → 很多 P3 是"噪音"
T+48h     ┃─ ⚡ 修复或降级/关闭 → 记录决策理由
```

#### 2.5.2 P3 处理策略

P3 级别通常是信息性的告警，处理策略：

- **每日汇总查看**即可，不需要逐条实时处理
- **定期清理**：每周回顾 P3 事件，关闭已自愈的、调整误报阈值
- **批量处理**：同类 P3 可以合并处理（如统一调整某个阈值）
- **不需要 Post-Mortem**，除非发现隐藏的系统性问题

### 2.6 升级触发条件

#### 2.6.1 自动升级规则

```yaml
# 升级触发器配置
auto_escalation:
  P0_critical:
    # 响应超时升级
    ack_timeout:
      seconds: 300        # 5 分钟未 ACK
      action: escalate_to_secondary
      notification: "phone_call + im + email"

    # 修复超时升级
    resolve_timeout:
      seconds: 1800       # 30 分钟未解决
      action: escalate_to_l2_tech_lead
      notification: "im_all_channels + management_email"

    # 二次升级（L2 处理 30 分钟仍未解决）
    second_escalation:
      seconds: 3600       # 总共 60 分钟
      action: escalate_to_l3_manager
      notification: "phone_call_manager"

  P1_high:
    ack_timeout:
      seconds: 900        # 15 分钟
      action: escalate_to_secondary
    resolve_timeout:
      seconds: 7200       # 2 小时
      action: escalate_to_l2

  P2_medium:
    ack_timeout:
      seconds: 3600       # 1 小时
      action: send_reminder_only  # 仅提醒，不强制升级
    resolve_timeout:
      seconds: 28800      # 8 小时
      action: escalate_to_l1
```

#### 2.6.2 手动升级条件

除自动超时升级外，以下情况 Primary 应主动申请升级：

| 触发场景 | 升级目标 | 理由 |
|---------|---------|------|
| 事件超出个人技术能力范围 | L2 Tech Lead | 需要更深的专业知识 |
| 需要跨团队协调（如联系云厂商） | L3 Manager | 需要管理权限 |
| 事件可能引发公关/法律问题 | L4/L5 | 需要高层决策 |
| 同时处理 3 个以上 P0/P1 事件 | L1 Secondary | 个人带宽不足 |
| 涉及安全漏洞或数据泄露 | L4 + Security Team | 合规和安全要求 |
| 需要紧急变更（如回滚版本） | L2 Tech Lead | 变更审批权限 |

**升级请求格式**：

### 4.1 内部通信渠道定义

#### 4.1.1 渠道矩阵

| 紧急程度 | 渠道 | 目标受众 | 响应期望 | 适用场景 |
|---------|------|---------|---------|---------|
| **🔴 紧急** | 电话 + SMS | Primary | < 2 分钟接听 | P0 告警 / 全平台故障 |
| **🔴 紧急** | 电话 + SMS | Escalation Contact | < 5 分钟接听 | 升级通知 |
| **🟠 高优** | 即时消息 (钉钉/企微/Slack) | Primary + Secondary | < 10 分钟回应 | P1 告警 / 重要通知 |
| **🟡 普通** | Email | 全团队 | < 4 小时阅读 | P2/P3 告警 / 周报 / 公告 |
| **🟢 状态** | Status Page / Dashboard | 外部用户 + 内部 | 实时自助查询 | 服务状态公示 |
| **📢 广播** | IM 频道 @all | 全团队 | 在线即见 | 重大事件通报 |

#### 4.1.2 各渠道使用规范

**电话/SMS 使用规范**：
- **仅用于 P0 级别告警和升级通知**
- 通话内容精简：事件级别 + 一句话描述 + 需要做什么
- SMS 作为电话的补充（含事件链接）
- 严禁用于非紧急事务

**IM（钉钉/企微/Slack）使用规范**：
- P1 告警使用 @mention 确保 visibility
- 消息格式统一（参见下方模板）
- 避免在公开频道讨论敏感信息（密钥、用户数据）
- 事件处理过程中保持简洁更新（避免刷屏）

**Email 使用规范**：
- Subject 行格式: `[GlobalReach-{LEVEL}] {简短标题}`
- P2 以上事件邮件应在 1 小时内发出
- 日报/周报使用固定模板
- 附件大小限制 < 10MB

**Status Page 使用规范**：
- 仅更新用户可见的服务状态
- 不透露技术细节和内部信息
- 状态变更必须有时间戳
- 提供 subscribe 功能让用户主动订阅

#### 4.1.3 事件状态更新频率

| 事件级别 | 更新频率 | 更新渠道 | 内容要求 |
|---------|---------|---------|---------|
| P0 | 每 10 分钟 | IM + Status Page | 当前状态 + 下一步 + ETA |
| P1 | 每 30 分钟 | IM + Email(每小时汇总) | 进展摘要 + 阻塞点 |
| P2 | 每 2 小时 | IM | 状态变化时才更新 |
| P3 | 每日汇总 | Email | 批量汇报 |

## 第五章：事后复盘流程

### 5.1 Post-Mortem 会议模板

#### 5.1.1 Blameless 文化原则

GlobalReach 的 Post-Mortem 遵循 **Blameless（无罪责）** 文化：

**核心原则**：
1. **人人都会犯错** — 关注系统和流程缺陷，而非个人过失
2. **复杂系统必然出故障** — 故障是设计的结果，不是人的失败
3. **从事故中学习比追责更有价值** — 惩罚会导致隐瞒，学习带来改善
4. **参与事故处理的人是最有价值的知识源** — 他们最了解发生了什么

**具体实践**：
- 会议中不允许使用指责性语言（"你的代码"、"你犯了错"）
- 关注"什么"、"怎么"、"为什么"，而不是"谁"
- 即使是人为失误，也要问"什么系统/流程让人容易犯错"
- 所有会议参与者（包括管理层）都遵守同样的规则

#### 5.1.2 会议组织

| 项目 | 要求 |
|------|------|
| **召开时机** | P0 事件: 24-72 小时内; P1 事件: 一周内 |
| **参会人员** | 事件处理者 + 相关组件负责人 + (可选) 利益相关者 |
| **会议时长** | 60-90 分钟（严格控制） |
| **会议主持人** | 非事件直接责任人（通常是 Tech Lead 或 SRE Lead） |
| **记录员** | 指定专人记录会议纪要 |
| **会前准备** | 所有参会人必须提前阅读事件 timeline |

### 5.3 行动项追踪机制

#### 5.3.1 行动项生命周期


#### 5.3.2 追踪工具和流程

| 工具 | 用途 | 更新频率 |
|------|------|---------|
| Incident System (API) | 记录行动项及其状态 | 实时更新 |
| 每周站会 | 过期和即将到期项的 review | 每周一次 |
| 月度回顾 | 完成率和质量的统计分析 | 每月一次 |

#### 5.3.3 行动项验收标准

每个行动项必须满足以下条件才能标记为"已完成":

1. **具体可测量**：有明确的 Done 定义（不是"优化性能"而是"P95 延迟降低到 <200ms"）
2. **有验证证据**：提供测试结果、监控截图、代码 PR 链接等
3. **经第三方确认**：至少一人（非责任人本人）确认完成
4. **有防回退手段**：关键措施需要有回归测试或监控告警

- **FMB 新增条目需要基于至少 2 次实际事件数据**（避免过度泛化）
- **季度审查**：每季度审查一次所有知识的准确性和完整性
