### 2.2 v6.0 核心设计哲学

```
v6.0 设计哲学 = Trae_IDE 五大原则 × 稳态运维实战 × 防幻觉防御

P1 第一性原理 (稳态版):
  不是"还有什么功能没做", 而是"如何让现有系统长期稳定运行"
  → 以系统稳定性为核心倒推所有运维决策

P2 复利效应最大化 (运维版):
  每次Session不仅完成任务, 还要产出可复用的运维资产和知识沉淀
  → 运维脚本/监控模板/SOP手册/经验教训 全部资产化

P3 自适应分层存储:
  Layer 1: 本协议 (~2000行) — 每次Session必读
  Layer 2: 防幻觉模板 — 每次断言前必查
  Layer 3: 无缝衔接指令 — 每次Session启停必用
  Layer 4: 运维资产库 — 动态增长
  Layer 5: 归档历史 — 极少访问

P4 自动化知识提炼:
  触发条件: 每次Session结束 / 发现新问题 / 解决故障
  → 自动产出: 经验教训 / 最佳实践 / 新方法论提案

P5 开放封闭原则 (稳态版):
  对扩展开放: 可无限添加新的监控规则/运维任务/安全策略
  对修改关闭: 已验证的核心架构保持稳定 (除非有充分理由)

P6 防幻觉防御 (新增):
  所有AI输出必须经过事实校验
  所有断言必须追溯到权威数据源
  所有变更必须符合宪法约束
```

#### Phase M 核心原则

```
Phase M 操作法则 = 最小变更原则 × 渐进式改进 × 可回滚保证

法则1: 最小变更原则
  每次Session只做一件事
  变更范围越小，风险越低，回滚越容易

法则2: 渐进式改进
  不追求一步到位
  每次只前进一小步，但方向正确

法则3: 可回滚保证
  每次变更都必须有明确的回滚方案
  如果无法回滚，则不执行该变更
```

## 八、回滚程序

### 8.1 通用回滚原则

```
回滚黄金法则 = 快速恢复 > 根因分析 > 完美修复

原则1: 时间优先
  系统不可用时，先恢复服务，再分析原因
  目标恢复时间(RTO): < 5分钟 (配置变更) / < 15分钟 (代码变更)

原则2: 版本回退优于热修复
  能用Git revert就用revert
  能用Docker rollback就用rollback
  尽量避免在压力下写"修复代码"

原则3: 回滚必须验证
  回滚不是目的，恢复可用才是
  回滚后必须执行完整的质量门禁检查

原则4: 记录一切
  回滚原因、步骤、结果、教训
  必须写入Session Report
```

### 8.2 分场景回滚方案

#### 场景A: Docker Compose配置变更回滚

```powershell
# 步骤1: 立即回退到上一个working版本
git checkout -- docker-compose.prod.yml

# 步骤2: 重启所有服务
docker compose -f docker-compose.prod.yml down
docker compose -f docker-compose.prod.yml up -d

# 步骤3: 等待30秒后验证
timeout /t 30
docker compose ps  # 确认所有容器Up

# 步骤4: 验证API健康
curl http://localhost:3000/api/v1/health
```

**RTO**: < 2分钟 | **复杂度**: 低

#### 场景B: Nginx配置变更回滚

```powershell
# 步骤1: 测试回退版本的配置语法
git checkout -- nginx/conf.d/production.conf
docker exec globalreach-nginx nginx -t

# 步骤2: 重新加载Nginx (不断服)
docker exec globalreach-nginx nginx -s reload

# 步骤3: 验证HTTPS访问
curl -k https://localhost/api/v1/health
```

**RTO**: < 1分钟 | **复杂度**: 低

#### 场景C: 应用代码变更回滚

```powershell
# 步骤1: 回退到最后一次commit
git revert HEAD --no-edit

# 步骤2: 重新构建并部署
docker compose -f docker-compose.prod.yml up -d --build api

# 步骤3: 等待健康检查通过
timeout /t 60
curl http://localhost:3000/api/v1/health
```

**RTO**: < 5分钟 (含构建) | **复杂度**: 中

#### 场景D: 数据库迁移回滚

```sql
-- 步骤1: 立即停止API容器 (防止新写入)
-- docker compose stop api

-- 步骤2: 从最近备份恢复
psql -U globalreach_user -d globalreach_prod -f backups/pg_latest.sql

-- 步骤3: 验证表结构完整性
\dt
SELECT count(*) FROM users;

-- 步骤4: 重启API
-- docker compose start api
```

**RTO**: < 10分钟 | **复杂度**: 高 | **前提**: 必须有有效备份

#### 场景E: 环境变量/Secrets变更回滚

```powershell
# 步骤1: 回退.env文件
git checkout -- .env.production

# 步骤2: 重启受影响的容器
docker compose -f docker-compose.prod.yml restart api postgres redis

# 步骤3: 验证连接
curl http://localhost:3000/api/v1/health
```

**RTO**: < 2分钟 | **复杂度**: 低

#### 场景F: 完整系统灾难恢复

```powershell
# 步骤1: 停止所有服务
docker compose -f docker-compose.prod.yml down

# 步骤2: 恢复Docker Volumes (如果有volume backup)
# docker volume restore ...

# 步骤3: 恢复数据库
psql -U globalreach_user -d globalreach_prod -f backups/pg_disaster_recovery.sql

# 步骤4: 恢复Redis (如果有关键数据)
redis-cli --rdb backups/redis_recovery.rdb

# 步骤5: 启动所有服务
docker compose -f docker-compose.prod.yml up -d

# 步骤6: 全面验证 (参考QG-01~QG-08)
```

**RTO**: < 30分钟 | **复杂度**: 极高 | **前提**: 必须有离线完整备份

### 8.3 回滚决策树

```
发现异常?
  │
  ├─ 影响范围? ─────────────────────────┐
  │   │                                 │
  │   ├─ 仅单个容器异常                  │
  │   │   └→ 重启该容器                  │

  │   │                                 │

  │   ├─ 数据库异常                     │

  │   │                                 │
  │   └─ 多组件同时异常/完全不可用      │
  │       └→ 场景F (灾难恢复)           │
  │
  └─ 回滚后仍异常?

```
